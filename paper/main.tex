\section{Related Work}
Natural User Interfaces (NUIs) have been a key focus in human-computer interaction (HCI) and gaming. Early systems heavily relied on dedicated hardware such as the Microsoft Kinect, which utilizes infrared depth sensors to capture skeletal data~\cite{b4}. While highly accurate for full-body tracking, Kinect requires significant physical space, specialized hardware, and creates high USB bandwidth overhead, making it impractical for lightweight consumer setups.

Another popular solution is the Leap Motion controller, a desktop-mounted device that uses infrared cameras and LEDs to extract fine-grained hand landmarks. Although Leap Motion provides exceptional precision and sub-millimeter accuracy specifically for hand gestures, it still requires proprietary external hardware and has a highly restricted interaction volume, which limits its widespread adoption in casual desktop gaming.

Alternatively, traditional computer vision methods using standard webcams have utilized color thresholding, background subtraction, and contour detection via libraries like OpenCV. However, these methods are highly sensitive to environmental factors such as skin color variations, background noise, and shifting illumination conditions, resulting in poor tracking robustness and high false activation rates in dynamic gameplay environments.

Recent advancements in deep learning have enabled robust, software-only landmark estimation using RGB webcams. Google's MediaPipe framework~\cite{b1} provides a real-time hand landmark tracking pipeline that estimates the $(x, y, z)$ coordinates of 21 keypoints from a single frame using an on-device machine learning model~\cite{b3}. The homuler plugin~\cite{b2} exposes this Hand Landmarker runner directly to the Unity game engine. 

While prior studies have evaluated MediaPipe's raw inference speed or accuracy in isolation~\cite{b5}, few have integrated it into a bimanual arcade game control system to evaluate the joint trade-offs of runtime performance (FPS, pipeline latency) and usability metrics (gesture accuracy, action latency). This paper addresses this gap by presenting a systematic evaluation inside the custom game ``Demon King vs. Rambo Frog''.

\subsection{Multi-Hand Landmark Extraction and Finger Tracking Mathematics}
Each hand yields 21 normalized landmarks $P_i=(x,y,z)$. Handedness tags distinguish left and right hands; labels are swapped when the webcam image is horizontally flipped.

Each finger's extension state is computed based on the vertical distance between its tip landmark $T_f$ and the proximal interphalangeal (PIP) joint landmark $P_f$. Let the normalized coordinate of a landmark be $V_j = (x_j, y_j, z_j)$ where $y_j$ is oriented downwards in MediaPipe's normalized space. For a finger $f \in \{\text{Index}, \text{Middle}, \text{Ring}, \text{Pinky}\}$, we define the tip landmark $T_f$ and PIP landmark $P_f$ as:
\begin{itemize}
    \item Index: $T_{\text{index}} = V_8$, $P_{\text{index}} = V_6$.
    \item Middle: $T_{\text{middle}} = V_{12}$, $P_{\text{middle}} = V_{10}$.
    \item Ring: $T_{\text{ring}} = V_{16}$, $P_{\text{ring}} = V_{14}$.
    \item Pinky: $T_{\text{pinky}} = V_{20}$, $P_{\text{pinky}} = V_{18}$.
\end{itemize}

The finger extension indicator function $E_f$ is defined as:
\begin{equation}
E_f = \begin{cases} 1 & \text{if } (y_{P_f} - y_{T_f}) > \delta \\ 0 & \text{otherwise} \end{cases}
\label{eq:finger_extension}
\end{equation}
where $\delta$ is the extension threshold (configured as $0.03$ in the Unity implementation). Note that because the $y$-axis points downwards, a smaller $y$-coordinate represents a higher physical position. Therefore, when the finger is extended upwards, $y_{P_f} - y_{T_f}$ yields a positive value.
The total number of extended fingers $N_{\text{extended}}$ is:
\begin{equation}
N_{\text{extended}} = \sum_{f} E_f
\label{eq:total_extended}
\end{equation}

\subsection{Continuous Coordinate Mapping, Aiming, and Locomotion Smoothing}
The raw hand landmarks are mapped continuously into Unity game inputs, and smoothing filters are applied to ensure control stability.

\textbf{Screen mapping:} Landmark~8 of the right hand is used for aiming. Its normalized coordinates $(x, y)$ are mapped to screen pixel coordinates $(X_{screen}, Y_{screen})$ via:
\begin{equation}
X_{screen}=x\cdot W_{screen}
\end{equation}
\begin{equation}
Y_{screen}=(1-y)\cdot H_{screen}
\end{equation}
where $W_{screen}$ and $H_{screen}$ are the screen width and height in pixels respectively. Note the vertical inversion $(1 - y)$ because MediaPipe's origin is at the top-left, while Unity's screen coordinate origin is at the bottom-left.

Then, the screen coordinates are mapped to 2D world space coordinates $\vec{P}_{world} = (X_{world}, Y_{world}, 0)$ via the main camera's projection:
\begin{equation}
\vec{P}_{world} = \text{ScreenToWorldPoint}(X_{screen}, Y_{screen}, -Z_{camera})
\label{eq:world_projection}
\end{equation}
where $Z_{camera}$ is the camera's depth Z-position. This coordinate determines the crosshair position and the rotation vector of the character's weapon.

\textbf{Locomotion smoothing:} To prevent sudden speed jerks or visual jittering of the character due to quick hand transitions or single-frame detection drops, the horizontal movement input $v_{smoothed}$ is smoothed using a linear transition towards the target velocity $v_{target}$:
\begin{equation}
v_{smoothed}^{(t)} = v_{smoothed}^{(t-1)} + \text{clamp}\left(v_{target} - v_{smoothed}^{(t-1)}, -\gamma \Delta t, \gamma \Delta t\right)
\label{eq:smoothing}
\end{equation}
where $\gamma$ is the velocity transition rate (configured as $8.0$), and $\Delta t$ is the time elapsed since the last frame.

\textbf{Bimanual control mapping logic:}
\begin{itemize}
\item \textbf{Left hand (Locomotion):}

\subsubsection{Performance vs.\ Usability Trade-off}
The results reveal a fundamental tension between runtime performance and the richness of gesture control. Enabling the MediaPipe CPU delegate with \texttt{numHands = 2} (S3) reduces average FPS from 60.0 to 34.8, a 42\% drop. However, lower FPS in S3--S4 does not automatically imply poor control quality. As long as $T_{\text{action}}$ remains below the 150\,ms threshold~\cite{b4} and GCA exceeds 85\%, the system remains playable for casual 2D arcade games despite sub-60 FPS on ultrabook hardware.

The mean end-to-end action latency of 158.6\,ms slightly exceeds the 150\,ms guideline, primarily due to the locomotion gestures G1 (211.2\,ms) and G2 (240.3\,ms) which have high $T_2$ (Mapping) components. This elevated $T_2$ is caused by the locomotion smoothing filter (Eq.~\ref{eq:smoothing}) which deliberately applies a gradual velocity transition rate $\gamma = 8.0$ to prevent jittering. The aim (G4) and shoot (G5) commands, being discrete threshold evaluations not subject to this smoothing, exhibit significantly lower latency at 77.7\,ms and 122.9\,ms respectively, both comfortably within the responsiveness threshold.

\subsubsection{Occlusion and Field-of-View Limitations}
One of the most significant sources of gesture misclassification in this system is partial or full hand occlusion. Two primary occlusion scenarios were identified during testing:

\textbf{Self-occlusion:} During rapid transitions between gestures, fingers may temporarily overlap from the camera's perspective, causing the finger count $N_{\text{extended}}$ to be momentarily under-estimated. For example, transitioning from the move-right gesture (2+ fingers extended) to the jump gesture (4 fingers extended) may cause a transient one-finger reading, briefly triggering a move-left command. This is particularly common when the player's hand rotates at an acute angle relative to the webcam.

\textbf{Frame boundary occlusion:} In S3 and S4, when the player simultaneously holds a jump gesture with the left hand and executes a shoot gesture with the right hand, the right hand naturally moves upward and may leave the camera frame. This reduces the Hand Detection Rate (HDR) from 94.8\% (S3) to 78.2\% (S4), directly increasing the False Activation Rate (FAR) as the system falls back to no-input state for the affected hand.

A practical mitigation strategy would be to add a temporal hold-on-last-result buffer: upon losing the hand for 2--3 consecutive frames, the system maintains the most recently detected gesture state rather than immediately resetting. This would reduce the perceptual disruption of brief tracking losses during fast gameplay.

\subsubsection{Environmental Lighting Sensitivity}
MediaPipe's Hand Landmarker model~\cite{b3} is designed to be robust across a variety of lighting conditions; however, the experiments in S4 (stress scenario) demonstrated measurable degradation under challenging environments. In conditions with low ambient light (below approximately 50 lux), the camera's automatic gain control increases sensor noise, which reduces the quality of hand edge features available to the neural network. This manifests as flickering landmark positions, reduced confidence scores, and occasional hand detection failures.

The overall HDR dropped from 95.2\% in S2 to 78.2\% in S4. The GCA for aim (G4) dropped from 100\% to 85\%, and for shoot (G5) from 100\% to 80\% under stress conditions. These degradations confirm that environmental lighting is a primary controllable variable for maintaining tracking quality.

Potential countermeasures include: (1) enforcing a minimum confidence threshold of 0.7 (up from the current 0.5) to filter out low-quality detections at the cost of slightly higher detection failure rate; (2) adding a pre-processing brightness normalization step to the RGBA32 texture before it is passed to the inference pipeline; (3) providing the player with on-screen feedback when hand tracking quality falls below a configurable threshold, prompting them to adjust their position or lighting.

\subsubsection{Ergonomics and Human Factors}
Beyond computational performance, the physical demands placed on the player represent a key usability consideration for any gesture-based game control system. Unlike keyboard-and-mouse input, which allows the player to rest their hands on a desk surface with minimal sustained muscle effort, controlling ``Demon King vs. Rambo Frog'' via hand tracking requires the player to hold their hands elevated in front of the camera for the duration of the gameplay session.

Informal observation during testing indicated that sustained arm elevation begins to cause noticeable fatigue in the shoulder and upper arm muscles after approximately 5--10 minutes of continuous gameplay. This fatigue manifests as involuntary micro-tremors which increase jitter in the detected landmarks, reducing the effective precision of the aiming system (G4) and increasing the likelihood of false activations.

Several design strategies can mitigate these ergonomic concerns. First, the camera could be mounted above the monitor so that the player only needs to raise their hands to a comfortable table-height level rather than holding them at face level. Second, the game's level design could incorporate natural pause moments (such as loading screens or dialogue sequences) where the player can lower their arms. Third, the gesture detection thresholds ($\delta$ and the move deadzone) could be adaptively adjusted to accommodate a player's gradually declining gesture amplitude as fatigue sets in, ensuring continued playability throughout an extended session.

\subsubsection{Comparison with Keyboard-and-Mouse Baseline}
The keyboard-and-mouse baseline (S1) provides a performance ceiling against which the hand-tracking scenarios can be compared. At 60\,FPS and effectively zero-latency input (keyboard polling occurs every frame), the baseline represents ideal conditions. The hand-tracking system under dual-hand mode (S3) achieves 34.8\,FPS and 158.6\,ms mean action latency. While this represents a significant overhead compared to the baseline, the system successfully demonstrates that natural gesture control of a real 2D arcade game is feasible on integrated-GPU consumer hardware without any external sensors or controllers. The trade-off is a reduction in frame rate and an increase in latency, both of which remain within ranges that preserve basic playability and player engagement for a casual arcade title.

\section{Conclusion and Future Work}
This paper presented a comprehensive performance and usability evaluation of continuous bimanual hand tracking via MediaPipe for real-time character control in the custom Unity 2D arcade game ``Demon King vs. Rambo Frog''. The system maps left-hand finger-count gestures to locomotion (move left, move right, jump) and right-hand gestures to combat actions (aim, shoot), using only a standard built-in webcam and CPU-based inference on an ultrabook with integrated graphics.

Across four evaluation scenarios, the system achieved an overall gesture command accuracy (GCA) of 96.5\% under normal conditions (S2 and S3), with a mean end-to-end action latency of 158.6\,ms. The dual-hand tracking configuration reduced frame rate by 42\% relative to the keyboard baseline, settling at 34.8\,FPS on the test machine. Despite these overheads, all five gesture commands remained reliably recognizable, and the system successfully controlled all in-game actions, confirming that MediaPipe-based bimanual input is a viable alternative to traditional hardware interfaces for casual 2D game interaction.

The evaluation also identified key limitations: the locomotion smoothing filter introduces elevated $T_2$ for movement gestures; hand detection reliability degrades under poor lighting (HDR drops to 78.2\% in the stress scenario); and sustained arm elevation causes player fatigue after extended play sessions.

Future work will explore five directions: (1)~testing standalone builds with the GPU delegate to reduce $T_1$ inference time; (2)~implementing a Kalman filter for landmark smoothing to reduce jitter and improve $T_2$; (3)~deploying automated gesture trial recording to reduce human observer error in GCA measurement; (4)~adopting adaptive confidence thresholds that adjust dynamically to ambient lighting conditions to improve S4 HDR; (5)~conducting a formal user study to quantitatively measure player fatigue, preference, and game performance between hand-tracking and keyboard-and-mouse control modes.
